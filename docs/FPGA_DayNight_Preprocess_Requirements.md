# 室外昼夜滤光 / 预处理需求说明（给硬件同学）

> **项目**：昇腾 310B + 紫光同创 PG2L100H + Unity 文旅互动  
> **用途**：整理需求，方便与皓耀杯（Intel 板）同学交流；同时明确「换板子有没有出入」  
> **最后更新**：2026-07-15

---

## 1. 我们用的是什么板？

| 角色 | 型号 | 干什么 |
|------|------|--------|
| 边缘 AI | 昇腾 310B4 | 跑视觉/语音/动作识别模型 |
| FPGA | **紫光同创 Logos-2 PG2L100H** | 确定性实时逻辑、前端预处理、事件融合（详见 `FPGA_AV_EventFusion.md`） |
| 表现层 | PC + Unity | 熊大数字人 / 3D 地图 |

**结论先说：**  
昼夜滤光适合放在 **FPGA 图像前端预处理**，不要做成「把整个 CNN 从 GPU 搬到 FPGA」。后者正是对方同学说的两大难点之一，和我们当前阶段目标不对齐。

---

## 2. 和 Intel 板（皓耀杯同学）有没有出入？

**有出入，但出入的是「怎么烧进去」，不是「算法能不能想」。**

| 维度 | 可共享（出入小） | 不可直接照搬（出入大） |
|------|------------------|------------------------|
| 算法思路 | 亮度判定、直方图、增益/对比度、Gamma、简单降噪 | — |
| RTL 逻辑 | 流水线、状态机、定点数公式 | — |
| HDL 语言 | Verilog / 对方用 Chisel 生成的通用逻辑可借思路 | Chisel 工程默认对接 Quartus，不能直接当紫光工程 |
| 开发工具 | — | Intel：**Quartus**；紫光：**PDS（Pango Design Suite）** |
| 约束/管脚/IP | — | 时钟、IO、UART、FIFO、图像接口 IP **全部重写** |
| 板级外设 | — | 摄像头接口、DDR、UART 电平、按键灯等 **按原理图重做** |
| 资源上限 | — | LUT/BRAM/DSP 数量不同，复杂模块可能要砍 |

### 给对方同学的话术（可直接复制）

> 我们板是紫光同创 **PG2L100H**，工具是 **PDS**，不是 Quartus。  
> 需要的是 **室外昼夜图像前端预处理**（判昼夜 + 按模式调亮暗），不是完整网络加速。  
> 你可以帮我们看：算法公式、流水线结构、定点化参数、用 Chisel/Verilog 写的**纯逻辑模块**。  
> 工具链、管脚约束、板级 IP 我们按紫光自己接；你那边 Intel 工程文件基本不能原样搬。

### 和对方两大难点的关系

对方提到的：

1. **GPU 模型参数 → FPGA 参数转换（少掉精度）**  
2. **用 Chisel 写硬件加速器**

对我们昼夜滤光 MVP：

- **第 1 点：暂时不需要。** 判昼夜 + 调亮暗主要是 ISP 类固定运算，不是搬 ResNet/YOLO。
- **第 2 点：思路可借，工程不可直接用。** 用 Verilog 写简单流水线即可；若对方愿意给 Chisel 生成的 Verilog 核，我们再进 PDS 适配。

## 3. 业务场景（为什么要做）

- 场景：**室外**乐园/文旅互动
- 摄像头接在 **310B**（不是 FPGA 直接接 USB 摄像头）
- 白天过曝、夜间偏暗，都会让视觉识别更不稳定
- 目标：在送进 310B 识别前，先做一层 **稳定、可解释、可答辩量化** 的前置处理

推荐数据流（与现有架构一致）：

```text
摄像头 → 310B 取帧
         ├─(可选) 旁路/下采样像素或统计量 → FPGA：昼/夜判定 + 滤波参数
         └─ 310B 按 FPGA 下发的模式做软件 ISP，或 FPGA 对ROI/统计量处理后回传模式事件
         → 识别模型 → Agent → Unity
```

**第一版建议（最简单可控）：**

```text
310B 算每帧平均亮度 / 简单直方图
  → UART 把亮度统计发给 FPGA
  → FPGA 状态机输出：DAY / NIGHT / TRANSITION + 增益/Gamma 档位
  → 310B 按档位做软件亮度校正（或只改推理阈值）
  → 稳定模式变化时再发事件给 Agent（如「已切换夜间模式」）
```

这样 FPGA 做的是 **实时判定 + 调度**，不用第一版就上完整像素流水线，联调最快。

**第二版（有余力再加）：** FPGA 对真正像素流做增益/Gamma（需稳定图像接口，不做 USB Host）。

## 4. 「昼夜滤光」产品需求（我们要的结果）

### 4.1 必须做到（P0）

| ID | 需求 | 验收标准（人话） |
|----|------|------------------|
| DN-01 | 自动判断当前偏白天还是偏黑夜 | 室外实测：白天大多判 DAY，夜间大多判 NIGHT，切换有防抖 |
| DN-02 | 夜间提高可见度（增益/Gamma/对比度其一即可） | 同场景夜间，处理后人眼更清晰；识别准确率相对未处理有提升或至少不明显掉 |
| DN-03 | 白天抑制过亮（限幅/降增益） | 强光下不过度爆白到脸/轮廓全丢 |
| DN-04 | 模式切换防抖 | 树影、逆光晃一下不要疯狂 DAY↔NIGHT 跳；如连续 N 帧或 T 毫秒才切换 |
| DN-05 | 对外可观测 | 可上报当前模式 + 亮度均值；网页/KPI 能显示（答辩用） |

### 4.2 最好有（P1）

| ID | 需求 | 说明 |
|----|------|------|
| DN-06 | 黄昏过渡档 TRANSITION | 不要只有两个极端档 |
| DN-07 | 与现有视觉事件融合 | 夜间适当提高「有人出现」置信阈值，减少误触发 |
| DN-08 | A/B 对比数据 | 开/关预处理各录一段，比误触发、漏检 |

### 4.3 明确不做（第一版）

| 不做 | 原因 |
|------|------|
| FPGA 跑完整检测网络 | 与 310B 定位重复，且落入「参数转换」大坑 |
| FPGA 直接接 USB 摄像头 | 已有设计文档结论：UVC Host 过重 |
| 光学硬件换滤光片 | 当前是数字滤光/ISP，不是机械滤光片 |

## 6. 建议给 FPGA / 算法的接口草案（给工程师）

### 6.1 输入（310B → FPGA）

| 字段 | 含义 | 示例 |
|------|------|------|
| `luma_mean` | 帧平均亮度 0–255 | 42 |
| `luma_p10` / `luma_p90` | 可选：暗部/亮部分位数 | 18 / 200 |
| `timestamp_ms` | 时间戳 | … |
| `frame_id` | 帧号 | … |

### 6.2 输出（FPGA → 310B / PC）

| 字段 | 含义 | 取值 |
|------|------|------|
| `illum_mode` | 光照模式 | `0=DAY, 1=NIGHT, 2=TRANSITION` |
| `gain_q8` | 定点增益 | 如 1.0 = 256 |
| `gamma_idx` | Gamma 档位 | 0/1/2 |
| `stable` | 是否已防抖稳定 | 0/1 |

可并入现有 **EdgeEvent** 扩展一类：`ILLUM_MODE_CHANGED`。

### 6.3 判定逻辑（可让对方帮忙调参）

伪规则（第一版够用）：

```text
若 luma_mean < T_night  连续 hold_ms → NIGHT
若 luma_mean > T_day    连续 hold_ms → DAY
否则 → TRANSITION
NIGHT 时 gain 提高一档；DAY 时 gain=1 并做高光限幅
```

参数 `T_night / T_day / hold_ms` 做成可配置寄存器，方便室外实测标定。

## 7. 和现有 FPGA 总方案的关系

本需求是 `FPGA_AV_EventFusion.md` 的 **图像前端扩展（文档中的可选项方向）**，优先级建议：

1. 先完成事件融合黄金模型（已规划）
2. **并行**：昼夜模式判定（软件模拟 → 再上 FPGA 状态机）
3. 有余力再做像素级 ISP 流水线

这样答辩可以说：

> FPGA 负责室外光照自适应与多模态事件的确定性融合；310B 负责感知；二者异构协同。

## 9. 关于「直方图方法」的评价（2026-07-15）

**总评：方向对，适合本项目；但建议「用直方图做统计判定」，慎用「整图直方图均衡化」当主滤光。**

| 做法 | 是否推荐 | 原因 |
|------|----------|------|
| 用直方图/均值/分位数判 DAY/NIGHT | ✅ 推荐 | 实现简单、FPGA 友好、可解释、答辩好讲 |
| 全局直方图均衡化（equalizeHist）再送模型 | ⚠️ 慎用 | 易放大噪声、改变对比分布，AI 模型未必受益 |
| CLAHE（分块自适应）轻度提亮暗区 | ✅ 可作夜间可选 | 比全局均衡稳，但要限对比度、做切换防抖 |
| 仅按模式调增益/Gamma | ✅ 第一版首选 | 最简单可控，和直方图判定天然配套 |

常见问题（室外 + 识别场景）：

1. **逆光/半边亮半边暗**：整图均值会骗人；至少加 `p10/p90` 分位数或天空/地面分区。
2. **全局均衡把噪声和树叶纹理一并拉高**：夜间提亮后「看起来更清楚」，但检测框可能更抖。
3. **摄像头自动曝光已在做事**：软件再猛拉一次可能打架；直方图判定后应「轻调」或改阈值，不要每帧用力均衡。
4. **模式来回跳**：树影晃一下直方图就变；必须有 hold 时间窗。
5. **训练分布偏移**：若模型在正常曝光图上训，重度 HE/CLAHE 后等于换了一种光照，可能掉点。

推荐落地（与 §3 一致）：

```text
直方图只出统计量 → 判模式 → 夜间小增益/温和 CLAHE；白天限幅
不要默认每帧 equalizeHist
```

## 10. 改进与风险（事后反思）

| 风险 | 说明 | 缓解 |
|------|------|------|
| 逆光/树影导致误判 | 均值不够稳 | 加分位数 + 防抖时间窗 |
| 全局 HE 伤识别 | 噪声与分布被拉歪 | 改增益/Gamma 或轻度 CLAHE |
| 对方 Chisel 工程无法上紫光 | 工具链不同 | 只要公式与 Verilog 模块，不取工程文件 |
| 范围做成「全网络加速」 | 周期翻倍 | 严格锁 P0 为 ISP/模式机 |
| 与事件融合抢工期 | 两边都重 | 昼夜先用 Python 模拟同一接口 |

## 11. 可拿去问硬件同学的「滤光细节」清单

> 前提跟对方说清楚：**我们板是紫光 PG2L100H，神经网络在 310B；FPGA 做 ISP/状态机，不做完整 CNN。**

### A. 判定昼夜（优先问）

1. 用亮度均值就够，还是必须要完整 256-bin 直方图？
2. 阈值怎么定？`T_night` / `T_day` 建议范围？要不要按场景标定？
3. 逆光怎么处理？要不要用 p10/p90 分位数，还是天空/地面分区？
4. 防抖：连续多少帧或多少毫秒才允许 DAY↔NIGHT 切换？
5. 要不要黄昏过渡档 TRANSITION，还是只有两档？

### B. 调亮/压暗（优先问）

6. 夜间用什么更稳：增益、Gamma、还是轻度 CLAHE？
7. 增益建议多大（例如 1.2～1.8）？会不会把噪声一起放大？
8. 白天过曝怎么压：限幅、降增益、还是压高光分位数？
9. 处理做在 **Y 通道** 还是 RGB 三个通道都做？
10. 定点怎么算（Q8/Q16）？查表还是乘法？FPGA 资源大概多少？

### C. 和 AI / 系统怎么配合（一定要问）

11. 预处理后的图再送检测，是否可能掉点？怎么做开/关 A/B 对比？
12. 摄像头自动曝光还开着时，软件/FPGA 应该「轻调」还是改成手动曝光？
13. FPGA 输出建议是：模式枚举 + 增益档位，还是直接改完整像素流？
14. 和 310B 之间传什么最小字段最合适（均值、直方图压缩、模式命令）？
15. 帧率掉了怎么办？预处理是否允许隔帧做？

### D. 对方能帮到哪一步（对齐期望）

16. 你能不能给：公式 + 伪代码 + 定点参数表？
17. 能不能给一段与工具无关的 Verilog（或不依赖 Intel IP 的逻辑）？
18. Quartus 工程 / Chisel 工程我们接不了，请不要以「能直接烧紫光」为前提。
